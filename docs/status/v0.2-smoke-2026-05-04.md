## What the smoke covered

End-to-end install ‚Üí verify ‚Üí uninstall against a tmpdir-fake `$HOME`.
Exact reproduction:

```sh
npm run build
SMOKE_HOME=$(mktemp -d /tmp/crew-smoke-XXXXXX)
CREW_LOG_LEVEL=info HOME=$SMOKE_HOME node dist/index.js install --target codex
CREW_LOG_LEVEL=info HOME=$SMOKE_HOME node dist/index.js verify
CREW_LOG_LEVEL=info HOME=$SMOKE_HOME node dist/index.js uninstall --target codex
rm -rf $SMOKE_HOME
```

### Confirmed working

- **`crew install --target codex`** writes:
  - `~/.codex/config.toml` containing a clean `[mcp_servers.crew]`
    block with `command = "<absolute node path>"` and
    `args = ["<absolute dist/index.js>", "serve"]`.
  - `~/.codex/prompts/crew.md` containing the rendered skill (body
    + tool list inserted at `{{TOOL_LIST}}`).
  - `~/.crew/install.json` schemaVersion=1 with the codex entry,
    absolute paths, ISO `installedAt`, and the resolved
    `serverCommand` + `serverArgs`.
- **`crew verify`** reads the manifest, finds the codex entry,
  scans the rendered skill for `mcp__crew__*` references, confirms
  every catalog tool is referenced, confirms no extras, confirms
  the host config still contains `[mcp_servers.crew]`, prints
  `crew verify: Codex ‚úì`.
- **`crew uninstall --target codex`** removes the skill file
  (`~/.codex/prompts/crew.md` gone), strips the `[mcp_servers.crew]`
  block from `config.toml` (file remains as a 0-byte file ‚Äî see
  Findings #3), and removes the codex entry from `install.json`.
- **TOML escaping** handled: absolute path `/Users/chasen/...` round-trips
  correctly through merge/remove.
- **Idempotency**: Running install twice produces the same config and
  skill bytes (asserted in unit tests; confirmed visually here).
- **The Claude Code session this doc is being authored in already
  has the `crew` skill registered** (visible in this session's
  available-skills list) ‚Äî meaning a previous install against the
  real `$HOME` did succeed against Claude Code's
  `~/.claude/skills/crew/SKILL.md`.

## Findings

### 1. Default log level silences install/verify/uninstall ‚Äî FIXED

The default console log level was `error`
(`src/utils/logger.ts:resolveInitialConsoleLevel`), so `crew install`
produced zero output on success. **Fixed:** default bumped to `info`.
The lifecycle commands now print confirmation lines like
`crew install: Codex ‚úì`. Set `CREW_LOG_LEVEL=error` to restore the
v0.1 silent-by-default behavior. `crew serve`'s logger still goes to
stderr per `src/cli/commands/serve.ts`, so the MCP wire protocol's
stdout discipline is unaffected.

Smoke confirmed: with no env vars set, `node dist/index.js install
--target codex` prints `[<ts>] INFO  crew install: Codex ‚úì`.

### 2. HTML comment leaks into the rendered skill ‚Äî FIXED

`skills/crew-captain.body.md` opened with an HTML comment block
(`<!-- Canonical skill body. Per-host templates wrap... -->`)
intended for repo readers ‚Äî it documents the body's provenance and
the PRODUCT_VISION editing rules. The renderer copied the body
verbatim, so this comment ended up in every rendered skill the host
CLI loaded.

**Fixed:** `loadCanonicalBody` now runs `stripHtmlComments` before
returning. The rendered skill is comment-free; the source body
keeps its provenance documentation. Multi-line comments are handled;
3+ newlines collapse to a paragraph break to keep the markdown clean.

Test added in `test/install/skill-renderer.test.ts` asserting the
rendered skill contains neither `<!--` nor `-->`.

Smoke confirmed: rendered codex skill goes directly from the codex
template framing to `## Crew ‚Äî orchestration playbook`, no comment
block in between.

### 3. Uninstall leaves an empty `config.toml`

When the only thing in `~/.codex/config.toml` was the crew block,
uninstall correctly removes the block but leaves a 0-byte file in
its place. Not a bug ‚Äî Codex tolerates empty TOML and we shouldn't
delete a config file the user might have expected to exist ‚Äî but
worth noting.

If a future cleanup pass wants to be extra-tidy, the install command
could remember whether it created the config file and let uninstall
delete only files it owns. Out of scope for v0.2.

### 4. `--target all` with no detected hosts is silent

`crew install --target all` skips hosts whose binary isn't on PATH.
Combined with #1, the user sees absolutely nothing ‚Äî no
"installed nothing because no host CLIs detected." Fixing #1 fixes
the worst of this; we could additionally print a one-line summary
at the end of `installCommand` (`installed: codex; skipped: claude-code (binary-not-found), gemini (binary-not-found)`).

  + integration tests cover every layer except the host CLI's MCP

## Carry-forward ‚Äî what M4 needs to assume

- The MCP server, skill, and install plumbing are stable surface.
- The 6-tool catalog (list_agents, run_agent, continue_run, merge_run,
  discard_run, get_run_status) is the v0.2 surface; M4 should not
  expand it.
- The skill body is the lever for behavioral A/B testing ‚Äî empty-skill
  control is "render with `{{BODY}}` replaced by a one-line "do nothing
  unusual" placeholder."
- Async fallback (60 s) and run-state persistence are real features
  worth probing in the eval (do hosts handle `status: "running"`
  gracefully? Do users actually use `get_run_status`?).
- Findings 1 and 2 are fixed. Findings 3 and 4 are minor and
  deferred ‚Äî they don't block M4 dogfooding.
