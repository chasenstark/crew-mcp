## What the smoke covered

End-to-end install → verify → uninstall against a tmpdir-fake `$HOME`.
Exact reproduction:

```sh
npm run build
SMOKE_HOME=$(mktemp -d /tmp/crew-smoke-XXXXXX)
CREW_LOG_LEVEL=info HOME=$SMOKE_HOME node dist/index.js install --target codex
CREW_LOG_LEVEL=info HOME=$SMOKE_HOME node dist/index.js verify
CREW_LOG_LEVEL=info HOME=$SMOKE_HOME node dist/index.js uninstall --target codex
rm -rf $SMOKE_HOME
```

### Confirmed working

- **`crew install --target codex`** writes:
  - `~/.codex/config.toml` containing a clean `[mcp_servers.crew]`

  - `~/.codex/prompts/crew.md` containing the rendered skill (body

## Findings

### 1. Default log level silences install/verify/uninstall — FIXED

The default console log level was `error`
(`src/utils/logger.ts:resolveInitialConsoleLevel`), so `crew install`
produced zero output on success. **Fixed:** default bumped to `info`.
The lifecycle commands now print confirmation lines like
`crew install: Codex ✓`. Set `CREW_LOG_LEVEL=error` to restore the
v0.1 silent-by-default behavior. `crew serve`'s logger still goes to
stderr per `src/cli/commands/serve.ts`, so the MCP wire protocol's
stdout discipline is unaffected.

Smoke confirmed: with no env vars set, `node dist/index.js install
--target codex` prints `[<ts>] INFO  crew install: Codex ✓`.

### 2. HTML comment leaks into the rendered skill — FIXED

`skills/crew-captain.body.md` opened with an HTML comment block
(`<!-- Canonical skill body. Per-host templates wrap... -->`)
intended for repo readers — it documents the body's provenance and
the PRODUCT_VISION editing rules. The renderer copied the body
verbatim, so this comment ended up in every rendered skill the host
CLI loaded.

**Fixed:** `loadCanonicalBody` now runs `stripHtmlComments` before
returning. The rendered skill is comment-free; the source body
keeps its provenance documentation. Multi-line comments are handled;
3+ newlines collapse to a paragraph break to keep the markdown clean.

Test added in `test/install/skill-renderer.test.ts` asserting the
rendered skill contains neither `<!--` nor `-->`.

Smoke confirmed: rendered codex skill goes directly from the codex
template framing to `## Crew — orchestration playbook`, no comment
block in between.

### 3. Uninstall leaves an empty `config.toml`

When the only thing in `~/.codex/config.toml` was the crew block,
uninstall correctly removes the block but leaves a 0-byte file in
its place. Not a bug — Codex tolerates empty TOML and we shouldn't
delete a config file the user might have expected to exist — but
worth noting.

If a future cleanup pass wants to be extra-tidy, the install command
could remember whether it created the config file and let uninstall
delete only files it owns. Out of scope for v0.2.

### 4. `--target all` with no detected hosts is silent

`crew install --target all` skips hosts whose binary isn't on PATH.
Combined with #1, the user sees absolutely nothing — no
"installed nothing because no host CLIs detected." Fixing #1 fixes
the worst of this; we could additionally print a one-line summary
at the end of `installCommand` (`installed: codex; skipped: claude-code (binary-not-found), gemini (binary-not-found)`).

### 5. Codex skill path + invocation are wrong — surfaced by real-host smoke

PRODUCT_VISION's host-trigger table assumed Codex loads skills at
`~/.codex/prompts/<name>.md` and triggers them with `/crew` or
`@crew`. Real-host smoke (Codex 0.128.0) showed neither holds:

- `/crew` returns `Unknown command: /crew. Did you mean /new?`
- The skill at `~/.codex/prompts/crew.md` does not appear in
  Codex's `### Available skills` list inside a session.
- Codex's own diagnostic (when asked why) explained that its skill
  system loads from "local SKILL.md" entries — a different format
  and a different path than what we install.

The MCP block at `[mcp_servers.crew]` in `~/.codex/config.toml` is
correct and Codex sees the server (`codex mcp list` reports `crew`
with status `enabled`). The `mcp__crew__*` tools are reachable when
called directly, so the substrate works — only the skill is
discoverable-as-a-skill.

Without the skill loaded, Codex doesn't have the captain playbook
biasing it toward MCP tools, so when asked "use crew to ..." it
falls back to whatever `crew` shell binary it can find on PATH (or
in CWD's `dist/`). That's how we ended up watching Codex shell out
to v0.1 commands during the smoke.

**Fix (M3 follow-up):**

1. Discover Codex's actual skill path + format. Likely
   `~/.codex/skills/<name>/SKILL.md` with frontmatter (mirroring
   Claude Code's convention) — confirm against Codex's own docs or
   release notes.
2. Update `src/install/hosts/codex.ts:skillPath` to the real path.
3. Update `skills/targets/codex.md.tmpl` to match the expected
   format (probably needs `name` + `description` frontmatter like
   `claude-code.md.tmpl` already has).
4. Update PRODUCT_VISION's host-trigger table.
5. Decide what to do with the slash-prompt path: `/crew` doesn't
   work, but Codex may have a different prompts mechanism that
   does. If users want explicit invocation in addition to
   skill-style auto-load, ship both files; otherwise drop the
   prompts path entirely.

**Carry-forward into the eval design:** the v0.2 skill behaves
differently across hosts. Claude Code auto-matches via description;
Codex needs whatever-its-real-mechanism-is; Gemini is an extension
descriptor. The A/B harness in M4 can't assume "skill loads" is
binary across hosts — it has to verify per host and capture which
loaded.

This is exactly the kind of host-spec drift PRODUCT_VISION's open
question Q5 ("Skill drift across host CLI updates") flagged as a
watch item. M3 was right to ship; the missing piece is per-host
verification of the load mechanism, which only real-host smoke
could surface.

### 6. Host CLI prefers local `dist/index.js` over MCP — DOCUMENT

When Codex was started inside `/Users/chasen/Documents/Projects/crew/`
(the v0.1 archive repo) and the user asked it to "use crew", Codex
shelled out to the local `dist/index.js` (v0.1 binary) instead of
calling the `mcp__crew__*` tools — even though the MCP server was
connected and listed.

Cause is two factors stacking:

1. **Skill not loaded on Codex** (Finding 5) → Codex has no
   playbook telling it to prefer MCP tools.
2. **Local `dist/index.js` decoy** → with the same binary name
   resolvable from CWD, Codex pattern-matches "use crew" to the
   shell binary it can see.

Moving Codex out of the crew repo (or into a clean directory with
no `dist/index.js`) made the dispatch route through MCP correctly.

**Mitigations for v0.2:**

1. **Skill-body wording** — `skills/crew-captain.body.md` should
   explicitly tell the host CLI to prefer `mcp__crew__*` tools
   over any `crew` shell binary, regardless of CWD. One-line
   addition.
2. **README install note** — recommend installing crew-mcp from
   outside its own repo and running smokes from clean directories.

**Won't fix in v0.2:** there's no clean way to make a host CLI
prefer MCP over shell binaries with matching names; that's host-
internal routing. Mitigations above are good enough.

### 7. `.crew/` pollutes the host repo's git status — FIXED

After a real-host dispatch, the host repo's `git status` showed
crew's run-state files staged for commit:

```
new file:   .crew/runs/.meta/<runId>.json
new file:   .crew/runs/<runId>/events.log
new file:   .crew/runs/<runId>/state.json
new file:   .crew/runs/<runId>/worktree
```

Worse, git treated the `worktree/` subdirectories as **embedded
repos** (they ARE git working trees) and tried to add them as
submodule pointers — emitting the standard "you've added another
git repository inside your current repository" warning. A
`git add .` from inside the worker (or from the user) would have
baked crew's internal state into the user's branch via `merge_run`.

Root cause: nothing tells the host repo to ignore `.crew/`.

**Fixed** via auto-gitignore in `src/git/gitignore-guard.ts`:

- New `ensureCrewIgnored(repoRoot)` helper.
- Called from `WorktreeManager`'s constructor (the moment crew
  knows it's about to write to `.crew/`).
- Idempotent: detects existing coverage of `.crew/`, `.crew/runs/`,
  `/.crew/`, etc. before appending. Tolerates negation (`!`) and
  comments.
- Failure-tolerant: if the write fails (read-only fs, permission
  denied), warn-logs and continues — never crashes serve startup.

Test added at `test/git/gitignore-guard.test.ts` (16 cases) covering
empty repo, existing `.gitignore` without coverage, existing with
exact / leading-slash / broader-rule coverage, idempotency,
trailing-newline preservation, and comment / negation handling.

**Recovery for already-broken repos** (where the user already
staged `.crew/...` before the fix landed):

```sh
git restore --staged .crew/
git rm -r --cached .crew/ 2>/dev/null || true
echo '.crew/' >> .gitignore       # only if your repo doesn't have it
git add .gitignore
```

### 8. Run state persists indefinitely after terminal — DEFERRED

`state.json` + `events.log` survive `merge_run` and `discard_run`
by design (M2's `cleanupByRunId` is non-recursive so
`get_run_status` can still report on a discarded run). Over time
`.crew/runs/` accumulates dead state for every run ever dispatched.
After 50 dispatches, that's 50 dirs of scratch.

Not blocking — `.gitignore` keeps it out of git, and disk cost is
trivial. But a `crew prune` command (clean up terminal-state runs
older than X days, configurable retention) would be a nice quality-
of-life addition.

**Defer to v0.3.** Not a v0.2 blocker. M4 dogfooding will tell us
whether anyone actually misses it.

### 9. Codex uninstall left orphaned `[mcp_servers.crew.tools.*]` sub-blocks — FIXED

While running `crew uninstall --target codex` during the smoke,
Codex refused to launch on the next start with:

```
Error loading config.toml: invalid transport in `mcp_servers.crew`
```

The user's `~/.codex/config.toml` had three orphan blocks Codex
auto-creates when the user grants per-tool approvals during a
session:

```toml
[mcp_servers.crew.tools.list_agents]
approval_mode = "approve"
[mcp_servers.crew.tools.run_agent]
approval_mode = "approve"
[mcp_servers.crew.tools.get_run_status]
approval_mode = "approve"
```

Our v1 `removeMcpBlock` matched only the exact `[mcp_servers.crew]`
header, so it stripped the parent (which holds `command` + `args`)
but left the `tools.*` children referencing a section that no
longer existed. Codex saw the orphans, couldn't find the parent's
transport config, and refused to load.

**Fixed** in `src/install/hosts/codex.ts` via a new
`NAMESPACE_HEADER_RE` (`/^\[mcp_servers\.crew(?:\.[^\]]*)?\]…/`)
and `locateAllCrewNamespaceBlocks` helper that finds every block
whose header is in the crew namespace. `removeMcpBlock` now splices
all of them out in one pass; spans are walked in reverse so indices
stay valid as we mutate.

`mergeMcpBlock` is **unchanged**: it still only replaces
`[mcp_servers.crew]` exactly. That's the right call — the
`tools.*` approval-mode entries are user-set preferences (Codex
remembers approvals across sessions); a re-install shouldn't wipe
them. Only uninstall does, because uninstall means "remove all
crew-related state."

5 new test cases in `test/install/hosts/codex.test.ts`:

- removes auto-created `tools.*` sub-blocks alongside the parent
- preserves sister namespaces (`[mcp_servers.linear]`, `[mcp_servers.figma]`)
- removes orphaned children when the parent is already gone (the
  user's exact failure mode)
- does NOT match sibling names like `[mcp_servers.crew_extension]`
  (conservative — only the `crew` namespace, not anything that
  starts with `crew`)
- merge re-install preserves `tools.*` approval prefs verbatim

**Recovery for users hitting this in the field:**

```sh
cp ~/.codex/config.toml ~/.codex/config.toml.bak
# If your tools.* blocks are at the end of the file:
awk '/^\[mcp_servers\.crew\.tools\./{exit} {print}' \
  ~/.codex/config.toml.bak > ~/.codex/config.toml
# Otherwise: open in editor, delete every [mcp_servers.crew.tools.*]
# block manually.
```

Codex should launch after that. Re-running the patched `crew
uninstall --target codex` is also safe (idempotent).

## Carry-forward — what M4 needs to assume

- The MCP server, skill, and install plumbing are stable surface.
- The 6-tool catalog (list_agents, run_agent, continue_run, merge_run,
  discard_run, get_run_status) is the v0.2 surface; M4 should not
  expand it.
- The skill body is the lever for behavioral A/B testing — empty-skill
  control is "render with `{{BODY}}` replaced by a one-line "do nothing
  unusual" placeholder."
- Async fallback (60 s) and run-state persistence are real features
  worth probing in the eval (do hosts handle `status: "running"`
  gracefully? Do users actually use `get_run_status`?).
- Findings 1, 2, 7, and 9 are fixed. Findings 3, 4, and 8 are minor
  and deferred — they don't block M4 dogfooding.
- **Finding 5 is M3-follow-up work that should land before M4
  starts.** Codex skill loading is broken, which means the M4 A/B
  harness can't measure with-skill vs. empty-skill on Codex until
  the skill path + format are corrected. Claude Code is unaffected;
  Gemini status TBD.
- **Finding 6** (host CLI prefers local `dist/index.js` over MCP
  when the same binary name lives in CWD) is a documentation issue
  for v0.2 — captured in carry-forward; users running smokes from
  inside crew checkouts need to run from a different directory or
  rename the local binary. Skill body can mitigate by explicitly
  telling the host CLI to prefer `mcp__crew__*` tools over any
  shell `crew`. Worth a follow-up edit to
  `skills/crew-captain.body.md`.
