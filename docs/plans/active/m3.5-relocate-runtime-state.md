# M3.5 — Relocate runtime state out of the host repo

**Status:** shipped.
**Anchor commit(s):** see `git log` (this commit and follow-up).
**Predecessor:** M3 (shipped, see `docs/plans/active/m3-status.md`).
**Related:** `docs/status/v0.2-smoke-2026-05-04.md` Findings 7 + 8.

**Suite at landing:** 500 passed / 3 skipped / 0 failed across 47
files. Lint clean. Build clean (161 KB ESM, flat vs. M3 — the
gitignore-guard deletion offset the new crew-home + plan-required
test cases). Net diff: -86 lines, -2 source files (gitignore-guard
+ its 16-test file gone; crew-home + host-repo-clean test added).

## Goal

Move every byte crew writes during a run out of the host repo. The
host repo's working tree should be untouched by crew's existence —
no `.crew/` directory, no `.gitignore` modification, no embedded-repo
warnings, no run state visible to `git status`. All run-scoped state
(state.json, events.log, the worktree itself, metadata, locks) lives
under `~/.crew/runs/<runId>/` instead, alongside the existing
`~/.crew/install.json`.

This was the right design from the start. M2's run-scoped layout
landed inside the host repo because the legacy task-keyed code at
`src/git/worktree.ts:50` already wrote there and we copied the
convention. Real-host smoke (Finding 7) surfaced the cost: a
`git add .` inside a worktree would have committed crew internals
to the user's branch via `merge_run`. The shipped fix
(`src/git/gitignore-guard.ts` at `08b70c9`) papers over the
symptom; this milestone removes the cause.

## Why now

- M4 dogfooding is about to start. Every additional surface area
  inside the host repo is a place for the eval to surprise us.
- The gitignore-guard is conservatively scoped (warn-and-continue,
  16 unit tests) but it modifies a tracked file in every host repo
  on first dispatch. That's an ongoing diff against the user's
  conventional state for as long as we ship it.
- Finding 8 (`crew prune`, deferred to v0.3) becomes trivial once
  state is in one global directory rather than scattered across N
  host repos.
- The "embedded repo" git warning, which the gitignore-guard does
  not silence on its own (only `git add` is unsafe; the warning
  fires on bare `git status` too), disappears entirely once the
  worktree is no longer under the host repo's worktree.

## End state

After this milestone, given a host repo at `~/code/widget/`:

```
~/.crew/
├── install.json              # unchanged (M3)
└── runs/
    ├── .meta/<runId>.json    # (was .crew/runs/.meta/<runId>.json)
    ├── .locks/<runId>/       # (was .crew/runs/.locks/<runId>/)
    └── <runId>/
        ├── state.json        # (was .crew/runs/<runId>/state.json)
        ├── events.log        # (was .crew/runs/<runId>/events.log)
        └── worktree/         # (was .crew/runs/<runId>/worktree/)
            └── <git checkout of host-repo branch crew-run/...>

~/code/widget/                # host repo
├── .git/
│   └── worktrees/<name>/gitdir → points into ~/.crew/runs/<runId>/worktree/.git
├── src/...                   # totally clean — no .crew/ here
└── .gitignore                # whatever the user has; crew never touches it
```

`git status` inside the host repo shows only the user's own changes.
The worktree's branch (`crew-run/<token>-<suffix>`) still merges
back into the host's HEAD because we merge by branch name, not by
worktree path.

## Scope

**In:**

- Relocate `RunStateStore` base path from `<projectRoot>/.crew/runs/`
  to `<crewHome>/runs/`.
- Relocate `WorktreeManager`'s run-scoped paths (`runBasePath`,
  `runMetadataPath`, `runLockPath`) to `<crewHome>/runs/...`.
- Add `repoRoot: string` to `RunStateV1` (schemaVersion stays at 1
  — additive optional field; readers tolerate absence on legacy
  records, writers always populate going forward).
- Resolve `<crewHome>` once at process start (`os.homedir() + '.crew'`,
  with `CREW_HOME` env override for tests + pathological CI).
- Delete `src/git/gitignore-guard.ts` and its tests.
- Delete the `ensureCrewIgnored(projectRoot)` call in
  `WorktreeManager`'s constructor.
- Delete the `isRuntimeStatePath` filter in
  `WorktreeManager.statusChangedFiles` (`src/git/worktree.ts:546`)
  — the host repo no longer has `.crew/` to filter out.
- Update `get_run_status` to surface `repoRoot` so the captain can
  tell the user which host repo a run came from.
- Update PRODUCT_VISION's "where state lives" wording.
- Update README's quickstart if it mentions `.crew/` paths.

**Out:**

- Migrating in-flight runs from a v0.2.0-dev install. Pre-existing
  runs in `<host>/.crew/runs/` stay where they are; the runtime
  reads them from the new location only. If the user has runs
  mid-flight at upgrade, they're orphaned (and `discard_run` will
  no-op cleanly because the runId metadata isn't found). This is
  acceptable given v0.2 isn't released yet — only smoke testers
  are affected, and they can `rm -rf <host>/.crew/`.
- Auto-cleaning the `.crew/` line from host repo `.gitignore`s.
  The line is benign (the directory just won't exist anymore);
  surgical cleanup is more invasive than the line. Document a
  one-liner for users who want to tidy.
- The legacy task-keyed `WorktreeManager` API
  (`createWorktree`/`mergeWorktree`/`cleanupWorktree`, anchored at
  `src/git/worktree.ts:78-228`). It's only called from the v0.1
  merge.ts path that the v2 server doesn't use. Leave it pointed
  at `<projectRoot>/.crew/worktrees/` until a future cleanup
  deletes it entirely. Adding a second relocation would muddy the
  diff for no functional gain.
- Cross-project run listing UX. `crew list` (or any global runs
  surfacing) is a v0.3 concern; the data shape supports it once
  `repoRoot` lands in state.json, but no command wires it.
- `crew prune` (Finding 8). Same v0.3 deferral as before, just
  trivially easier to implement now.
- XDG_DATA_HOME / XDG_STATE_HOME on Linux. `~/.crew/` matches what
  M3 already established; cross-platform XDG support is a v0.3
  conversation.

## Migration for in-flight runs (v0.2.0-dev users only)

We aren't shipped yet, so the only affected users are smoke-testers
who already dispatched runs. After upgrading:

- Old runs at `<host>/.crew/runs/<runId>/` become invisible to
  `get_run_status` / `merge_run` / `discard_run`. The runtime
  reads `<crewHome>/runs/...` only.
- Recovery: `rm -rf <host>/.crew/` and re-dispatch the work.
  Nothing critical is lost — these were smoke-test runs by
  definition.
- Document in the M3.5 status note + the v0.2.0-dev release
  CHANGELOG entry.

If we wanted to be tidier (we don't, but for the record): a one-shot
`crew migrate` could walk every `<host>/.crew/runs/` it finds and
rewrite paths into `<crewHome>/runs/`. Costs more code than the
problem deserves at this stage.

## Risks + open questions

1. **`$HOME` writability in pathological CI.** The
   `install-manifest.ts` already assumes `~/.crew/` works (M3),
   so this milestone doesn't expand the assumption. `CREW_HOME`
   env override is the escape hatch.

2. **Worktree paths leak into git plumbing.** `git worktree list`
   in the host repo will show entries pointing into `~/.crew/...`.
   That's fine — it's the truth — but a curious user running
   `git worktree list` will see paths they don't recognize. The
   alternative (worktree under host repo) is what we're trying
   to leave behind. Document the path in `get_run_status`'s
   response so the captain can explain it on demand.

3. **`merge_run` from a different worktree.** Today the worktree
   is a sibling of the host repo on the same filesystem, which
   makes `git merge` cheap. Putting the worktree under
   `~/.crew/...` puts it on potentially-different filesystems
   (e.g., `~` on a different mount than `~/code/`). git handles
   this correctly — worktrees on remote-mounted filesystems
   work, just slower. Worth a smoke check on macOS where
   `~` is on the boot volume but project repos may live on
   external drives. Not a correctness concern, just performance.

4. **Symlink resolution.** If a host repo is itself a symlink
   (`~/code/widget` → `/Volumes/work/widget`), `process.cwd()`
   may resolve to either depending on how the user invoked
   their host CLI. We should record `realpath(repoRoot)` in
   `state.json` so two captures of the same repo via different
   symlinks share state. Add a one-line `realpath` call in
   `RunStateStore.create()`.

5. **Discoverability regression.** Users who learned to `ls
   .crew/runs/` in their project will now find nothing. Mitigation:
   `get_run_status` already returns the worktree path; the
   captain should mention it when the user asks "where's the
   run." Nothing structural to fix.

6. **Concurrent crew serves across host repos.** Two host CLIs
   (e.g., a Codex session in repo A and a Claude Code session in
   repo B) now share `~/.crew/runs/`. They never collide on
   runIds (UUIDs), and `repoRoot` disambiguates state.json
   entries, but the lock directory `~/.crew/runs/.locks/` is
   shared. The lock keying is by runId already, so this is
   safe. Worth a smoke check (run two host CLIs against
   different repos simultaneously) before declaring the
   milestone done.

## Decisions worth flagging now (before implementation)

1. **`~/.crew/` over XDG.** Matches the existing
   `~/.crew/install.json` location M3 established. XDG support is
   v0.3+. Linux users who really care can `CREW_HOME=$XDG_DATA_HOME/crew`.

2. **`repoRoot` is an additive field, not a v2 schema.** Schema
   stays at v1; readers tolerate absence; writers always populate.
   Avoids the migration plumbing M2 explicitly chose to defer.

3. **Don't migrate in-flight runs.** v0.2.0-dev only; no shipped
   users to break. A migration command costs more than the problem
   deserves at this stage.

4. **Leave the legacy task-keyed worktree path alone.**
   `<projectRoot>/.crew/worktrees/` (`worktree.ts:48-52`) is only
   touched by v0.1 merge.ts paths the v2 server doesn't reach.
   Migrating it would muddy the diff for zero functional gain;
   it can die when the v0.1 leftovers in `src/workflow/` are
   swept (already noted in IMPLEMENTATION_PLAN's "Lingering v0.1
   dead code" section).

5. **`realpath(repoRoot)`, not `repoRoot` raw.** Symlinked host
   repos shouldn't fragment state across multiple `repoRoot`
   values that point at the same underlying tree.
